# Can AI and Multi-Agent Systems Develop and Debug Robotics Software? An Evidence-Grounded Literature Review (2023–2026)

## TL;DR
- **The stigma is half-right and half-outdated.** The evidence supports a split verdict: AI/multi-agent systems are now genuinely strong at the *software-engineering* side of robotics — writing, repairing, and debugging ROS/ROS2 code, automated program repair, and architecture recovery — but remain weak as the runtime *embodied brain* for low-level sensorimotor control, spatial reasoning, multi-agent coordination, and error recovery. "AI can't do robotics" is increasingly wrong about *code* and increasingly right about *embodiment*.
- **Concrete numbers bear this out.** On general software repair, frontier agents resolve roughly 80% of SWE-bench Verified issues (Claude Opus 4.5 at 80.9%; vendor claims reach into the 90s) and multi-agent debuggers post state-of-the-art results on Defects4J/QuixBugs/Codeflaws. On robotics *software development*, closed-loop "draft-debug-improve" loops (RoboCoach) lifted policy-development success from 0.391 to 0.730 — above a human-authored reference — and a Claude-Agent-SDK system (FAEA) reached 84.9–96% on LIBERO/ManiSkill/MetaWorld with privileged state. But on embodied control, EmbodiedBench's best model (GPT-4o) scored only 28.9% on low-level manipulation, and Butter-Bench's best LLM scored 40% vs 95% for humans.
- **There is a real academic↔industry naming gap, and your three "mystery tools" resolve.** Peer-reviewed robotics work almost never names commercial tools — it uses GPT-4/Claude inside custom frameworks like ROSA. The names you gave are very likely speech-to-text artifacts: **"Open Cloud" → OpenClaw**, **"Nanocloud" → NVIDIA NemoClaw**, **"Arisem" → ARIS** (an agentic baseline benchmarked in robotics work). Claude Code and Harness are real and documented, but mostly in grey literature, not peer-reviewed robotics studies.

## Key Findings

1. **General software repair/debugging is a solved-enough problem at the frontier.** SWE-bench Verified (500 human-validated GitHub issues) went from GPT-4 solving ~1–2% in 2023 to frontier systems near 80% by 2026. Harder, contamination-resistant benchmarks (SWE-bench Pro) cut scores roughly in half, exposing real brittleness on unfamiliar codebases — but the trajectory is unambiguous.
2. **Robotics-specific code generation and debugging works and is improving fast,** with the strongest results coming from *agentic closed-loop iteration* (write → run → read error → fix), not single-shot generation. This is the single most important design lesson across the literature.
3. **Multi-agent role architectures (programmer/tester/debugger) help on decomposable tasks but add cost and can propagate errors.** They are not a free win.
4. **Using LLMs/VLMs as the robot's runtime brain remains the justified core of the stigma.** Low-level manipulation, spatial reasoning, coordination, and error recovery are documented, repeated failure modes across every embodied benchmark.
5. **The boundary is the whole point:** AI is good at *reasoning about and generating code that controls robots*, and weak at *being the controller*. Experiments should exploit the former and avoid the latter.

## Details

### A. The software-engineering cluster (primary focus)

**Program repair / debugging benchmarks.** SWE-bench (Jimenez et al., 2023) and its human-validated 500-task subset SWE-bench Verified (OpenAI, 2024) are the field standard. Progress has been steep: GPT-4 solved ~1–2% of tasks in standard prompting in late 2023; scaffolded agents reached 12–20% through 2024; top systems crossed 50% in mid-2024; and by 2026 the leaders sit near 80% (Claude Opus 4.5 reported at 80.9% on Verified; GPT-5.3 Codex reported at 85%; vendor "Claude Mythos" marketing figures reach 93.9–95.5%). These high numbers must be read with care: the field-wide average across many evaluated models is ~63%, and harder, contamination-resistant benchmarks tell a sobering story. **SWE-bench Pro** (Deng et al., 2025; 731 enterprise-grade tasks) sees Claude Sonnet 4.5 lead at only 43.6% Pass@1, with the held-out *commercial* set under 20%; the same Claude Opus 4.5 that scores 80.9% on Verified scores 45.9% on Pro under standardized scaffolding on tasks it could not have seen in training. Adversarial strengthening (**SWE-ABS**) drops scores by ~15–19 points. **Bottom line:** frontier agents are genuinely competent at real bug-fixing, but curated-benchmark success materially overstates production reliability.

**Multi-agent debuggers on classic benchmarks.** FixAgent / UniDebugger (Jin et al.; hierarchical multi-agent framework, EMNLP 2025) achieves a new SOTA on **Defects4J** (197 correct fixes, 286 plausible — beating ChatRepair by ~25%), fixes **all** bugs in **QuixBugs** across Java and Python, and generates 2.2× more plausible fixes than the best prior method on **Codeflaws** — notably without requiring prior fault localization. **RepairAgent** (Bouzenia et al., 2024) autonomously fixes 164 Defects4J bugs via a finite-state-machine tool controller, fixing 39 bugs no baseline could. **RGD** (Jin, Sun, Chen, 2024) — a three-LLM Guide/Debug/Feedback architecture — reports +9.8% on HumanEval and +16.2% on MBPP over prior SOTA. **CodeSim** (2025) adds simulation-driven planning/debugging without extra test generation. Collectively these establish that LLM agents are effective at fault localization and program repair on real Java/C/Python code.

**Multi-agent software-dev frameworks.** **ChatDev** (Qian et al., 2023; CEO/CTO/programmer/reviewer/tester roles) and **MetaGPT** (Hong et al., 2023; SOP-driven, 85.9%/87.7% Pass@1 on HumanEval/MBPP) demonstrated that decomposition beats single-shot generation. But the same literature documents the costs: MetaGPT's auto-generated tests were only ~80% accurate on HumanEval; large agent counts (MetaGPT 5, ChatDev 7) drive high token cost (often >$10 per HumanEval task) and risk **error propagation** when a weak tester/reviewer passes incorrect code. The design recommendation that emerges is adaptive role activation, not maximal agent counts.

**Robotics code specifically.** A cluster of ROS-focused systems exists: ROSGPT, ROS-LLM, OperateLLM (ReAct + DeepSeek Coder v2 + rclpy to create nodes/publishers), and NASA-JPL's **ROSA** (Royce et al., 2024; LangChain-based ReAct agent for ROS1/ROS2 inspection, diagnosis, and debugging), which is the de facto external baseline in the field. A 2026 paper (Benchat et al., Clausthal University) uses LLM-based agents to *recover hierarchical architecture* of ROS2 systems from code and launch files, reporting **high precision but recall that degrades with repository complexity** because of implicit launch-file semantics — a concrete, honest limitation for maintenance use cases.

**ROS debugging, quantified.** **ROS Help Desk** (Katuwandeniya & Widhanapathirana, CSIRO Robotics; arXiv 2507.07846; produced via the IEEE IES GenAI Hackathon) extends ROSA with proactive log monitoring, multimodal sensor diagnostics, and RAG over an evolving error database. Across **seven injected fault types** (lidar/image drop, delay, corrupt; node crash), each tested **five times**, it identified the correct error **100% of the time vs ROSA's 29%** — ROSA could not detect communication errors or node crashes at all. Scored against eight expert-defined criteria by an LLM (Claude) judge, it averaged **68%** overall, and three expert ROS users rated it **~4/5**. Critically, it identified the *true injected root cause* only **18%** of the time: this quantifies a key nuance — AI is far better at error *detection and explanation* than at *root-cause diagnosis*.

**Robotics policy *development* via agentic loops (the strongest direct evidence on your core question).** **RoboCoach / RoboCoach-Bench** (Lei, Liu, Zhang et al., Shanghai Jiao Tong University + Peking University; arXiv 2601.21570, "From Digital to Physical: Digital Agents as Autonomous Coaches for Physical Intelligence" — note: the task's working title "EmboCoach-Bench" maps to this paper). Across **32 tasks on four simulators** (ManiSkill3, RoboTwin2, Robomimic, MetaWorld) spanning RL and IL and architectures from MLPs/RNNs to ACT, Diffusion Policy, and VLAs, a dual-agent **"Draft-Debug-Improve"** loop (coding agent + feedback agent over tree-structured memory, with video diagnostics from a VLM) raised benchmark-wide mean success from **0.391 (non-agentic prompting) to 0.730 — above the human-authored reference of 0.602.** Per-model: Gemini 3.0 Pro **0.398→0.802 (+40.5 pp)**; GPT-5.2 **0.406→0.770 (+36.4 pp)**; Claude Opus 4.5 0.577→0.785; six of seven frontier models exceeded the human reference after closed-loop optimization. In a matched-harness comparison (GPT-5.2 fixed), RoboCoach reached **0.59 on a RoboTwin IL subset**, beating Codex (0.46), Claude Code (0.17), and non-agentic prompting (0.23); on a 45-task RoboTwin IL set it scored 0.59 vs **ARIS 0.38, OpenClaw 0.10, Hermes 0.08**. Real-robot transfer improved real-robot success **0.51→0.68** averaged over four tasks. Notably, **no model–task pair declined** under the agentic loop (60.5% improved, 39.5% unchanged). The reported mechanism is explicitly transferring software-engineering iterative debugging to robot-policy development.

**FAEA** ("Demonstration-Free Robotic Control via LLM Agents," Tsui, Fang, Hwu; arXiv 2601.20334) uses an *unmodified* **Claude Agent SDK** (Claude Opus 4.5) in a ReAct hypothesis-test-refine loop. With privileged ground-truth state it reached **84.9% (LIBERO), 85.7% (ManiSkill3), 96% (MetaWorld)**; one round of human feedback lifted LIBERO to **88.2%**. It frames the mechanism as "the same iterative reasoning that enables software agents to debug code." Caveats are significant: simulation-only, privileged state (not RGB), and precision-contact tasks still fail (PegInsertion 0%, PlugCharger 0–60%). It is also a self-described hobby project (unvetted preprint).

### B. The embodied contrast cluster (where the stigma holds)

**EmbodiedBench** (Yang et al., ICML 2025; arXiv 2502.09560; 1,128 tasks across EB-ALFRED, EB-Habitat, EB-Navigation, EB-Manipulation): "MLLMs excel at high-level tasks but struggle with low-level manipulation, with the best model, GPT-4o, scoring only 28.9% on average" (57.7% on EB-Navigation but 28.9% on EB-Manipulation). On high-level household tasks Claude-3.5-Sonnet led (64.0% EB-ALFRED, 68.0% EB-Habitat). This high-level/low-level cliff is the cleanest single illustration of the boundary.

**Butter-Bench** (Sharrock et al., Andon Labs, 2025; arXiv 2510.21860): the best LLM (Gemini 2.5 Pro) scored **40%** on real-robot practical-intelligence tasks vs a **95%** human mean; models ranked Gemini 2.5 Pro > Claude Opus 4.1 > GPT-5 > Gemini ER 1.5 > Grok 4. Embodied-reasoning fine-tuning did **not** improve scores. Failures concentrated in multi-step spatial planning and social understanding; Claude Sonnet 3.5 produced a now-famous "doom-spiral" while disoriented. Co-founder's verdict: "LLMs are not ready to be robots."

**RoboBench** (Luo et al., 2025; arXiv 2510.17801; 14 SOTA MLLMs as "embodied brain," ~6,000–7,000 QA pairs): fundamental limitations in implicit instruction comprehension, spatiotemporal reasoning, cross-scenario planning, fine-grained affordance understanding, and execution-failure diagnosis. Gemini-2.5-Pro led (62.96 perception reasoning) but trailed humans (74.30); implicit instructions dropped performance ~30%.

**PARTNR** (Chang et al., Meta, 2024; arXiv 2411.00081; 100,000 multi-agent household tasks): SOTA LLMs show "poor coordination and failures in task tracking and recovery from errors"; paired with real humans they needed 1.5× the steps of two humans collaborating. **Habitat-MAS / EMOS** (2024; arXiv 2410.22662): heterogeneous multi-robot coordination requiring embodiment-aware reasoning. **VLABench** (2024; arXiv 2412.18194): on long-horizon language-conditioned manipulation, both current SOTA VLAs and VLM-workflows struggle. Convergent spatial-reasoning evidence across **BEAR**, **RoCoBench**, and dedicated spatial-reasoning surveys shows LLMs handle shallow/metric spatial tasks but fail on combinatorial planning, orientation tracking (near-chance, strong directional priors), and multi-frame alignment — a representational mismatch between discrete tokens and continuous geometry.

### C. The named-tools hunt

- **Claude Code** (Anthropic): heavily used in robotics *grey literature*. Community ROS2 "skills"/templates exist (henkirobotics, arpitg1304/robotics-agent-skills, j3soon/ros2-essentials with CLAUDE.md, harunkurtdev/ros2-claude-code-template). Henki Robotics documented Claude Code (claude-opus-4-6) generating a ROS2 odom→speed package: with a best-practice CLAUDE.md it produced separated core logic, a custom RobotSpeed.msg, throttled logging, launch/param files, and unit tests. No peer-reviewed robotics benchmark of Claude Code itself, except as a *baseline* in RoboCoach where it scored low (0.17 on RoboTwin IL) — a useful, cautionary datapoint that generic coding agents underperform robotics-specialized agentic loops.
- **Harness** (harness.io): a real AI-native CI/CD/DevOps platform with autonomous "Harness Agents" (autofix, code review, code coverage, manifest/Zero-Day remediation) running as governed pipeline steps; its DevOps Agent runs on Claude Opus 4.6. **No documented robotics-specific use** — it is general SDLC automation. Beware terminology overlap: "harness engineering" is also a generic term for agent scaffolding (cf. OpenAI's "harness engineering" Codex case study, where a 3–7 person team shipped ~1M lines of agent-written code at 3.5 merged PRs/engineer/day — relevant as an analogy for disciplined agentic dev, not robotics).
- **"Open Cloud" → OpenClaw**: open-source autonomous agent framework by Peter Steinberger (Nov 2025; originally Clawd/Moltbot), model-agnostic with a skills system and MIT-licensed gateway. Reaches robotics via **AgenticROS** and **ROSClaw** (an OpenClaw ROS2 framework, arXiv 2603.26997, which explicitly characterizes LLM robot control as "flexible but still fragile"). Benchmarked in RoboCoach at 0.10 on RoboTwin IL. Documented security issues (CVE-2026-25253, CVSS 8.8).
- **"Nanocloud" → NVIDIA NemoClaw**: NVIDIA's open-source reference stack (announced GTC 2026) to run OpenClaw/Hermes agents safely inside OpenShell sandboxes, integrating Claude Code, Codex, and Cursor. Robotics relevance is *infrastructural* (secure, local always-on agents on DGX Spark/RTX), not a robotics capability result.
- **"Arisem" → most likely ARIS**: an agentic baseline benchmarked against RoboCoach (0.38 on the 45-task RoboTwin IL set), placing it above OpenClaw/Hermes but well below the closed-loop RoboCoach system. **Flag:** this is a probable speech-to-text artifact; if your intended reference was a process-mining tool ("ARIS AI Companion"), that is unrelated to robotics. Treat ARIS-the-agent-baseline as the relevant entity.

**Characterizing the gap:** the most rigorous peer-reviewed robotics work (ROSA, ROS Help Desk, EmbodiedBench, RoboBench) names *models* (GPT-4/4o, Claude, Gemini) inside *custom* LangChain/ReAct frameworks, not commercial products. Named commercial agentic tools (Claude Code, Codex, OpenClaw, Hermes, ARIS) appear almost exclusively in (a) the newest preprints as comparison baselines (RoboCoach) and (b) grey literature. If you want defensible evidence about a *specific named tool* in robotics, you will largely have to generate it yourself.

## Recommendations

**Stage 1 — Start where AI is strongest (software dev/debug).** Run a controlled experiment with a frontier agentic coding tool (Claude Code or Codex) on a real ROS2 codebase covering: (a) bug-fix tasks with known-good tests (build your own mini "ROS2-bench"), (b) a CLAUDE.md/AGENTS.md harness encoding ROS2 best practices (QoS, lifecycle nodes, launch semantics), and (c) a closed-loop run-test-fix loop. Compare single-shot vs agentic-loop vs multi-agent. **Threshold to proceed:** if the agentic loop resolves >50% of held-out ROS2 bugs with passing tests, the "AI can't debug robotics code" stigma is falsified for your setting (RoboCoach/RepairAgent/ROS Help Desk suggest this is achievable).

**Stage 2 — Test closed-loop policy development in sim.** Replicate a RoboCoach/FAEA-style draft-debug-improve loop in ManiSkill3 or LIBERO with privileged state first, then degrade to RGB. **Benchmark that changes the verdict:** if agentic iteration beats both non-agentic prompting *and* the human-authored reference (as RoboCoach reports a 0.39→0.73 lift), invest further. If precision-contact tasks stay near 0% (as in FAEA), treat fine sensorimotor control as out of scope for LLM orchestration.

**Stage 3 — Do NOT use LLMs as the runtime low-level controller.** Keep a hierarchical split: LLM/agent for high-level reasoning and *code generation*; classical/learned controllers (MoveIt2, Nav2, VLAs, RL policies) for sensorimotor execution. Every embodied benchmark (EmbodiedBench, Butter-Bench, RoboBench, PARTNR) makes this non-negotiable today.

**Cross-cutting instrumentation.** Log full agent trajectories; measure not just pass rate but cost/tokens, retries, and **test-gaming** (the SWE-bench Verified stub-the-test risk applies to ROS tests too). Use multi-agent roles only when tasks decompose cleanly, and watch for error propagation from a weak tester/reviewer. Where possible, prefer contamination-resistant or freshly-authored tasks to avoid training-data leakage inflating your results.

## Caveats
- **Evidentiary weight differs sharply.** SWE-bench/Defects4J/QuixBugs/Codeflaws results and EmbodiedBench (ICML 2025), Butter-Bench, RoboBench, PARTNR, VLABench are peer-reviewed or rigorously documented. RoboCoach (2601.21570) and FAEA (2601.20334) are very recent preprints — strong signal, not yet vetted; FAEA is a self-described hobby project. Vendor figures (e.g., "93.9–95.5% Claude Mythos") are marketing-adjacent and should be discounted relative to standardized, contamination-resistant leaderboards.
- **Benchmark inflation is real.** SWE-ABS shows ~15–19-point drops under adversarial strengthening, and SWE-bench Pro roughly halves Verified scores. Curated-benchmark success ≠ production reliability.
- **Privileged-state caveat.** FAEA's high numbers (84.9–96%) use ground-truth state, not raw vision; real-world RGB performance is materially lower, and all FAEA results are in simulation.
- **Detection vs root-causing.** ROS Help Desk detected faults 100% of the time but found the true root cause only 18% of the time — do not conflate error detection/explanation with diagnosis.
- **Fast-moving model landscape.** Sources reference GPT-5.x, Gemini 3.0 Pro, and Claude Opus 4.5/4.6 consistent with a 2026 date; treat exact leaderboard positions as perishable.
- **The academic↔industry gap** means the most current named-tool evidence lives in blogs/whitepapers; the only rigorous head-to-head of named tools in robotics found here is RoboCoach's baseline comparison (Codex/Claude Code/OpenClaw/ARIS/Hermes all underperforming a robotics-specialized closed-loop agent).
