## The two roles

A secure tunnel has two ends:

| End | Runs where | Token | Mode |
|-----|-----------|-------|------|
| **destination** | on the **TurtleBot Pi** (the thing being reached) | `destinationAccessToken` | `localproxy -d` |
| **source** | on **your laptop** (where you initiate the connection) | `sourceAccessToken` | `localproxy -s` |

## What AWS IoT Core is

A managed cloud service that lets **devices** (your TurtleBots) and the **cloud** talk securely, mostly over **MQTT** (a lightweight publish/subscribe messaging protocol). Think of it as a secure message broker in the cloud plus a set of features built on top.

The main bridge is the **Rules Engine** — but there are a few distinct paths. Let me lay them out.

## 1. The Rules Engine (the primary path)

This is how ~90% of IoT-Core-to-AWS integration happens. A rule has three parts:

**a) A SQL statement** that selects messages from MQTT topics:
```sql
SELECT temperature, battery FROM 'turtlebot468/telemetry' WHERE battery < 12
```
It runs on every message hitting that topic, can filter (`WHERE`), and reshape the payload.

**b) One or more "Actions"** — where the matched message goes. Each action targets a specific AWS service:

| Action → Service | What it does |
|---|---|
| **Lambda** | Run arbitrary code (the most flexible — Lambda can then call *anything*) |
| **DynamoDB** | Write the message to a NoSQL table |
| **S3** | Store the payload as an object |
| **SNS** | Send a notification (SMS/email/push) |
| **SQS** | Queue the message for a consumer |
| **Kinesis / Firehose** | Stream into analytics/data lakes |
| **CloudWatch** | Push a metric or alarm |
| **Timestream** | Write to a time-series database |
| **republish** | Send it back to *another* MQTT topic |
| **Step Functions** | Kick off a workflow |

**c) An IAM role** — the rule assumes a role granting it permission to write to that target service. (This is separate from the *device* policy; it's the *rule's* permission.)

So the flow is:
```
Device → MQTT topic → Rule (SQL filter) → Action → AWS service
```

Yes — and from EC2 you actually have an *extra*, easier option that physical devices don't: **IAM authentication**. Here's the breakdown.

## EC2 can talk to IoT Core three ways

**1. As an MQTT client using IAM (SigV4) — the easy way for AWS compute**
Because EC2 runs inside AWS, give the instance an **IAM role** (instance profile) and it can:
- **Publish** messages with a single API call — no certs needed:
  ```bash
  aws iot-data publish --topic 'turtlebot468/cmd' \
    --payload '{"action":"dock"}' --region us-west-2
  ```
- Connect over **MQTT-over-WebSocket** signed with the instance's IAM credentials.

The IAM role's permissions (e.g. `iot:Publish`, `iot:Connect`, `iot-data:*`) replace the device cert + IoT policy. This is the natural pattern for a backend app on EC2.

**2. As an MQTT client using X.509 certs — same as a device**
You *can* drop the same cert/key onto EC2 and connect exactly like the TurtleBot does (the SDK sample in your folder would run unchanged). But this is usually the wrong choice for EC2 — certs are for *devices*; IAM is for *AWS compute*.

**3. As a consumer of the data IoT Core forwards**
Often EC2 doesn't talk to IoT Core directly at all. The Rules Engine pushes messages into **SQS / DynamoDB / Kinesis**, and your EC2 app reads from those. This decouples the backend from the broker.

| Connecting from | Authenticates with |
|---|---|
| TurtleBot (device) | X.509 cert + IoT **policy** |
| EC2 / Lambda (AWS compute) | **IAM** role/credentials (SigV4) |
