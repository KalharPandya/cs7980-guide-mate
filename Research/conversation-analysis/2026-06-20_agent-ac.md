## Motive

The user wanted a deep research deliverable: enumerate the `depthai_ros_driver`
(ros-humble-depthai ~2.29.0) ROS 2 parameters that affect STABILITY and
AUTO-RECOVERY of an OAK-D-LITE camera on a Raspberry Pi 4, then return a markdown
parameter table (name, purpose, default, recommended-for-Pi4, why) plus prose on
the most stable, auto-recovering bring-up combination, with cited URLs
(file: 2026-06-20_agent-ac.jsonl, date: 2026-06-20, turn: 1).

This maps directly to a real hardware gotcha in the project: the OAK-D depth
runtime drop where the device drops to bootloader and the stock driver wedges
forever (CLAUDE.md gotcha 8). The research targets the `i_restart_on_diagnostics_error`
self-heal parameter and related stability knobs.

## Topics covered

- depthai_ros_driver v2 (humble branch) parameter names and exact defaults
  (file: 2026-06-20_agent-ac.jsonl, date: 2026-06-20, turn: 13, 18, 20, 23).
- The version mismatch between the v3/RVC4 docs and the v2 humble source, and
  resolving which defaults are authoritative (turn: 15, 20, 43).
- The driver's auto-recovery mechanism: `SystemLogger` diagnostics node,
  `Camera::diagCB()`, `i_restart_on_diagnostics_error`, and `Camera::restart()`
  doing stop+start (turn: 24, 25, 49 region, 43).
- Sensor-level (left/right/rgb/stereo) parameters: `i_fps`, `i_max_q_size`,
  `i_low_bandwidth`, resolution (turn: 23).
- USB speed, pipeline type, NN type, IMU enable, sync enable trade-offs for a
  power-constrained Pi4 (turn: 14, 20, 43).

## Tasks given

1. Research and enumerate every relevant stability/auto-recovery parameter for
   depthai_ros_driver on OAK-D-LITE / Pi4, using WebSearch and WebFetch loaded via
   ToolSearch (turn: 1).
2. Provide purpose, type, default, and recommended Pi4 value for each named
   parameter (camera-level and sensor-level) (turn: 1).
3. Identify the parameter(s) that make the node auto-recover after a device
   disconnect rather than sitting dead (turn: 1).
4. Return a markdown table plus prose on the most stable + auto-recovering
   bring-up combination, citing URLs (turn: 1).

## Tasks done

- Loaded WebSearch/WebFetch via ToolSearch and ran two web searches plus multiple
  source fetches (turn: 3, 6, 7, 11, 12).
- Pulled the authoritative v2 humble defaults directly from the source files
  `camera_param_handler.cpp`, `sensor_param_handler.cpp`, and `camera.cpp`
  (turn: 12, 16, 21, 22, 23).
- Caught and corrected a version mismatch: recognized the docs were v3/RVC4 and
  that the project uses v2 humble, then verified the v2 defaults against source
  (`i_usb_speed=SUPER`, `i_pipeline_type=RGBD`, `i_nn_type=spatial`,
  `i_restart_on_diagnostics_error=false`), explicitly noting `i_nn_type` is
  `spatial` in source, not `none` as the searches suggested (turn: 15, 20, 43).
- Reconstructed the recovery path end to end: diagnostics SystemLogger ->
  `diagCB()` watches for error status -> if `i_restart_on_diagnostics_error: true`
  calls `restart()` (stop+start, re-init device); noted there is no separate
  USB-disconnect callback and no watchdog timer (turn: 24, 25, 43).
- Delivered the final research report with a parameter table and prose on the
  stable, auto-recovering bring-up recipe (turn: 43).
